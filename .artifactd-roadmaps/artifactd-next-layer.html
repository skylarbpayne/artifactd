artifactd roadmap · 2026-05-18

The next layer is where this gets unfairly useful.

My take: do not turn artifactd into a general app platform. Make it a safe visual command surface: artifacts can hold lightweight browser state, request a small set of approved server actions, age out cleanly, and render project dashboards from Skyvault + Kanban + entity graph context.

Recommendation

Capability-based interactivity

Support client-side JS now, then server actions only through explicit, named capabilities. No arbitrary backend code from generated artifacts. That way Echo can build visual workflows without accidentally creating a public RCE machine. Tiny detail.

Cleanup

Archive, don’t delete

Add lifecycle metadata and hide old artifacts by default. Delete only after explicit approval or a conservative retention rule for smoke tests.

Dashboard

Project cockpit generator

Generate HTV and wedding dashboards from notes, Kanban, artifacts, people/entities, and source links. The dashboard is the interface; Skyvault/Kanban stay truth.

1. Interactivity model

Ship first

Allow normal HTML/CSS/JS inside artifact files. This supports filters, tabs, checkboxes, calculators, drag sorting, local annotations, and review state via localStorage.

  • Great for Jacqueline/Echo presentation review boards.
  • Great for daily cockpit checklists.
  • Great for project dashboards with filters and expand/collapse.
Truth boundary

Browser-local state is convenience, not truth. If a decision matters, the artifact should offer a deliberate “record this” action that writes to Kanban/Skyvault through a controlled endpoint.

Capability registry

Artifact metadata declares allowed capabilities: kanban.comment, kanban.create_task, artifact.archive, artifact.describe, later draft.email.

Action endpoint

POST to /{slug}/_actions/{capability}. artifactd checks password cookie, capability allowlist, payload schema, size limits, and audit log.

Approval gates

External sends, publishing, scheduling, payments, credentials, repo pushes, or destructive changes become queued approval requests, not direct buttons.

  • No arbitrary Python/Node/server-side artifact code. Ever, unless we build a sandbox later.
  • Protected artifacts required for anything with action capabilities or private project context.
  • CSRF token per artifact session, payload schemas, audit table, rate limits.
  • Every action writes an append-only receipt: who/when/slug/capability/payload hash/result.
  • Dangerous operations become approval cards for Skylar/Jacqueline, not clickable “oops” buttons.

2. Archive / cleanup

Artifact soup is real. The home page will get useless unless lifecycle is first-class.

metadata additions: - status: active | archived | deleted - project: htv | wedding | echo | palmer | misc - tags: [] - owner: palmer | echo | skylar | jacqueline - source_refs: note paths, kanban ids, urls - expires_at - archived_at - archive_reason
  • Home hides archived by default.
  • /archive shows archived artifacts with restore.
  • CLI: artifactd archive <slug>, restore, prune --dry-run.
  • Suggested auto-archive: smoke tests 14d, daily cockpits 90d, review boards 180d unless pinned.
  • Never delete protected/private artifacts automatically at first. Archive only.

3. Project dashboards

Build a generator that turns entity graph + Skyvault + Kanban into a readable cockpit.

Inputs
  • Project note frontmatter + live work sections
  • Kanban tasks/comments/status
  • CRM/entity notes for people/orgs
  • Artifact metadata links
  • Optional Gmail/Calendar/Drive evidence when explicitly requested
Output
  • Status, blockers, owner map, next actions
  • Timeline/deadlines
  • People/entity graph
  • Evidence links
  • Safe action buttons: create task, comment, mark reviewed, request approval

Other high-leverage ideas

  • Artifact lineage: “this dashboard was generated from these notes/tasks/emails at this time.” Prevents stale pretty dashboards from lying.
  • Review packets: one-click shareable packet for “Jacqueline, review these 3 decisions,” with links and visual options.
  • Staleness badges: red/yellow/green based on source age and unresolved blockers.
  • Artifact inbox: artifacts needing human review, grouped by person/project.
  • Decision capture: buttons that record “approved / rejected / needs changes” as Kanban comments, not just local checkbox state.
  • Template catalog: presentation storyboard, website before/after, event run-of-show, project cockpit, vendor decision board.
  • Mobile-first mode: because a lot of personal/admin review happens on the couch, not in a terminal like a gremlin.

Generated as an artifact because this is exactly the kind of thing chat is bad at holding. Source truth stays in Skyvault/Kanban; this is the review surface.
